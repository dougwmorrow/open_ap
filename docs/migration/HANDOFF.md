## §1. Mission

UDM (Unified Data Management) pipeline: extract from Oracle (DNA) and SQL Server (CCM, EPICOR) sources, tokenize PII, write Parquet snapshots to a network drive, promote through SCD2 in SQL Server Bronze, mirror to Snowflake for analytics. Audit-grade traceability for financial regulatory compliance.

**The single outcome that matters**: see `NORTH_STAR.md`. Every design choice serves it.

**Pipeline shape (greenfield deployment, not migration)**:

```
Oracle / SQL Server (source)
    ↓ Polars + ConnectorX extract
Python pipeline on RHEL Linux (PII tokenization, hash, SCD2 promote)
    ↓
[Network Drive Parquet]    [SQL Server Bronze SCD2]   [General.ops metadata]
            ↓                          ↓
       Snowflake Iceberg ←── Snowflake Bronze mirror
                              (analytics + reconciliation)
```

## §8. Pitfalls — don't repeat these

These are real mistakes the project made; reading them saves you from repeating.

1. **Producing artifacts without validation.** Round 1 v1 had 8 🔴 bugs that shipped to "complete" because no validation step ran. Now D55 + D56 enforce the gates.
2. **Premature 🟢 lock.** v2 had 3 🔴 the producer (me) missed. The validation agent caught them. Independent review is non-negotiable.
3. **Decision text drifting from doc body.** D45.6's v2 expansion claimed in the schema doc body but not in `03_DECISIONS.md`. Cross-doc consistency is Gate 1.
4. **Filtered-index + lookup-filter mismatch.** SP-1 v2 used `Status='active'` filter against an unfiltered UNIQUE index → THROW on every benign re-tokenization. Always cross-check that index predicate matches lookup predicate.
5. **Fixing one bug introduces another.** v2 SP-1 fix interacted badly with the D45.6 Status-flip pattern. **First-pass validation** caught it — which is exactly why D55 (5-gate discipline) is mandatory before any 🟢 lock; D56 (second-pass) is the safety net for the *next* iteration's regression risk.
6. **Plan drift from execution.** ~50 decisions, ~120 edge cases, ~zero lines of production code → Round 0.5 spike exists to validate decision combinations before they ossify (D47).
7. **Cross-doc inconsistency from over-eager decision capture.** Round count differed across `02_PHASES.md`, `phase1/00_phase_overview.md`, `PHASE_1_DEEP_DIVE_PLAN.md`. Sweep these whenever you change phase scope.
8. **Risk-delta-without-register-update.** When a decision's risk-delta line claims `🆕 NEW: R<N>`, `✅ MITIGATED: R<N>`, `⬆️ ESCALATED: R<N>`, or `⬇️ DE-ESCALATED: R<N>`, ALWAYS verify the corresponding `RISKS.md` row exists / is updated BEFORE locking. This pattern bit three rounds in a row: D49 v2→v3 (P2 status not flipped in `04_EDGE_CASES.md`), D61 strict-mode (R12 mitigated-claim without RISKS.md update), and D62 dog-food (R16 declared without RISKS.md row). Each was caught by a first-pass validator under D55+D56. The discipline works — but writing the decision-and-register update *together* would skip the cycle entirely.
9. **Fix-introduces-fresh-instance-of-same-bug-class.** When a 🔴 fix introduces a NEW reference to a canonical source, the validator (whether first-pass producer self-check or post-fix second-pass agent) MUST re-verify EVERY new reference against canonical. **Seven consecutive evidence-rounds** confirm this pattern is structural, not coincidental — by Round 4 the cumulative count reached 19 fresh-instance occurrences across 8 sub-classes.
