## §4 — Deeper insights (THE differentiator vs SESSION_RESUME.md)

### Architectural decisions made

**Decision: Phase 1 vs Phase 2 scope split for udm-session-compactor**
- Alternatives considered: (a) build full token-tracking subsystem upfront; (b) Phase 1 manual-trigger first then Phase 2
- Why-not full upfront: Claude Code hooks lack `CLAUDE_CONTEXT_USED` access (active feature request `anthropics/claude-code#34340`). Workaround requires parsing `~/.claude/projects/<slug>/<session-uuid>.jsonl` + Token Counting API subprocess invocations (~500ms-2s latency per measurement; rate-limited at higher tiers). Build investment ~2-3 days for unclear workflow value.
- Why Phase 1 first: validate workflow before instrumentation. Phase 1 builds the SNAPSHOT FORMAT + COMPOSITION + DISCIPLINE; Phase 2 only adds TRIGGERING automation.
- Reversal trigger: 3+ Phase 1 invocations + user signal "I wish this auto-triggered when context fills up" OR actual context-compaction event causes downstream agent confusion that a Phase 2 trigger would have prevented.

**Decision: 4-layer false-positive prevention architecture**
- Alternatives considered: (a) monolithic patch on each new check vs (b) layered architecture
- Why-not monolithic: every false positive class would need bespoke patch. Pattern accumulation risk over time.
- Why layered: each layer addresses a structurally distinct failure mode without coupling.
  - Layer 1 (WARN-severity): heuristic checks degrade gracefully on false positives (no BLOCK)
  - Layer 2 (shared helper `_is_empirical_anchor_context()` B-488): suppress self-reference meta-pattern (check fires on its own closure commit's citations)
  - Layer 3 (Mechanism A Step 6 B-490): reviewer-methodology forward-prevention (regex-completeness verification against ≥2 canonical format variants)
  - Layer 4 (`_false_positive_log.md` accumulation tracker B-489): per-event accumulation discipline + round-aggregate classification trigger
- Evidence: 4 false-positive events in session arc (FP-1 through FP-4) confirmed the architectural pattern works — every FP class fits cleanly into one of the 4 layers without overlap.

**Decision: SUBSTRATE_EDIT clause requires PRE-COMMIT independent reviewer for skill SKILL.md edits**
- Alternatives considered: (a) inline-self-review for skill authoring vs (b) spawn reviewer per substrate-edit
- Why-not inline: commit `0a0ff49` (B-316 closure with TEST-only) precedent showed substrate-edit inline-self-review missed 2 🔴 BLOCK findings retroactively. Producer judgment "this is just an internal refactor" is exactly the failure mode.
- Why spawn reviewer: substrate IS the discipline; breaking discipline silently is the failure mode. Mechanism C-1 git hooks now enforce this mechanically (`tools/cascade_classifier.py::is_substrate_path()` + `tools/check_commit_msg.py` cascade-evidence check).
- Empirical: 2 retroactive cascades 2026-05-16 both found 🔴 BLOCK findings in substrate refactors → motivated the substrate-edit clause formalization at B-317 Phase 1B.

**Decision: 1-event-vs-2-event empirical threshold for B-N opens**
- Convention: open NEW B-N at 1st event observation; structural formalization (HANDOFF §8 Pitfall sub-class addition; new shared helper; new discipline rule) waits for 2nd event per HANDOFF §8 convention.
- Why: balances eager-tracking (don't lose pattern observations) with restrained-architecture (don't formalize on coincidence).
- Empirical: B-488 was 2-event closure (B-480 + B-487 absorbed); B-491 deferred at 1-event with explicit defer-to-2nd-event citation. Pitfall #9.o was formalized at 5-event, then strengthened across instances 6-9 because producer-judgment-based discipline structurally vulnerable.
