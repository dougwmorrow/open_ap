### §4.1 — Architectural decisions + rejected alternatives

**Decision 1: D125 3rd-value `'both'` extension over hard-cutover** (the foundational architectural call this session)

- **Decision**: Extend D63's 2-value CDCMode CHECK constraint to 3 values: `('change_detect', 'parquet_snapshot', 'both')`
- **Alternatives considered**:
  - (a) Keep D63 2-value; direct cutover — **REJECTED**: HIGH Bronze risk on first D2 cutover; no validation period; no shadow-write capture
  - (b) Add 3rd value `'both'` (CHOSEN) — incremental safety; forward-only-additive extension of D63; preserves D63 defaults for existing tables
  - (c) Add 3 sub-variants of `'both'` immediately (`'both_legacy_feeds'`, `'both_parquet_feeds'`, `'both_compare'`) — REJECTED for v1: scope creep; only ONE sub-variant needed; future enhancement enabled by D92
  - (d) Runtime env-var or CLI flag — REJECTED: per-table flexibility needed (different tables migrate at different times); D63 architecture already chose per-table column
- **Why-not for (a)**: Direct flip skips the 30-day shadow-write validation period that is canonical migration safety per Google SRE Workbook (Chapter 13 selective reprocessing) + AWS Well-Architected Operational Excellence Pillar
- **What would trigger reversal**: empirical operator-feedback that the `'both'` mode added I/O cost (~30%) is intolerable on production-scale tables AND no false-positive parity findings occur during shadow period

**Decision 2: BOTH_LEGACY_FEEDS sub-variant for `'both'` v1**

- **Decision**: When `CDCMode='both'`, legacy CDC drives Bronze; Parquet write is shadow/audit substrate
- **Alternatives considered**:
  - BOTH_PARQUET_FEEDS — REJECTED for v1: higher Bronze-impact risk; Parquet path unproven in production
  - BOTH_COMPARE — DEFERRED to future enhancement: requires parity-check tool first (B-545); requires SCD2 input adapter for replay path (B-552)
- **Why-not**: minimize Bronze risk during migration; legacy CDC is canonical production path with months of testing. Future BOTH_COMPARE enabled by B-555 closure + new `'both_compare'` CHECK value (forward-only-additive per D92).

**Decision 3: B-544 v1 partial-closure pattern (NotImplementedError for `'parquet_snapshot'` mode)**

- **Decision**: Implement `'change_detect'` + `'both'` modes; `'parquet_snapshot'` mode raises NotImplementedError citing B-552 (new B-N opened immediately at v1 closure)
- **Alternatives considered**:
  - Full 3-mode in one commit — REJECTED: cdc_result adapter for replay path + delete-detection mechanism are substantial separate work
  - `'parquet_snapshot'` falls back to legacy CDC silently — REJECTED: hides operator intent + bypasses D125 plan §5.2 pseudocode
- **Why-not**: scope-minimization for tractable cohort; immediate-B-N-open mirrors B-552 visibility (vs B-545 v1's initial "track as new B-N if/when needed" deferral, which was corrected to explicit B-555 open at remediation `1995aa3`)

**Decision 4: `classify_parity()` asymmetric semantic at B-553 closure** (Parquet < Bronze = MAJOR_DRIFT regardless of %)

- **Decision**: Add CRITICAL-class guard at top of `classify_parity()`: `if parquet_count < bronze_count: return MAJOR_DRIFT`
- **Rationale**: Parquet is source-exact per D115; Bronze is a subset after CDC NULL-PK filter (P0-4). Parquet < Bronze indicates data loss between extraction + Parquet write — CRITICAL operational signal, NOT a percentage-tolerable drift.
- **Alternative considered**: Maintain symmetric drift logic; let operator interpret. REJECTED: defeats the purpose of automated verdicts; operator might dismiss as "small drift, ignore"
- **Trade-off accepted**: 999/1000 = MAJOR_DRIFT but 1000/999 = CLEAN; intentional asymmetry surfaces in test parametrize with explanatory comments

**Decision 5: Tool-local SQL resolvers over TableConfigLoader import at B-554**

- **Decision**: Author `_resolve_bronze_table_name()` + `_resolve_pk_columns()` as tool-local SQL helpers in `tools/validate_parquet_vs_stage.py` rather than importing `orchestration.table_config::TableConfigLoader`
- **Rationale**: TableConfigLoader pulls polars + connectorx + 30+ UdmTablesList columns; tool only needs StripSuffix + BronzeTableName + PK columns. Lightweight tool-local SQL avoids dependency surface.
- **Trade-off**: Future TableConfig refactor that changes the bronze-name composition would silently NOT propagate to the tool. B-557 expansion candidate if 3rd consumer appears.
- **Forward-prevention**: tool's docstring + B-554 closure body cite `TableConfig.bronze_full_table_name` as the canonical reference

**Decision 6: CLAUDE.md Do-NOT rule LIFTED format (first instance precedent set at `00039a1`)**

- **Decision**: Lifted Do-NOT rules use `~~strikethrough~~` body + **LIFTED YYYY-MM-DD** annotation + B-N closure citations + empirical-anchor preservation (rather than deleting the rule entirely)
- **Rationale**: preserves audit trail of why the rule existed + what closed it; future-maintainer reading CLAUDE.md sees the historical context
- **Trade-off**: accumulates strikethrough cruft if many rules get lifted over time; might warrant periodic POLISH_QUEUE-driven archive at round close-out
