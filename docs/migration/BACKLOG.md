# Backlog

WSJF-prioritized 🟡 follow-ups across the project. Single source of truth for "small things that didn't block locking but need to land before phase close."

## Methodology

Each item gets:
- **Cost of Delay (COD)**: 1-5 (5 = waiting hurts a lot; 1 = waiting is harmless)
- **Job Size (JS)**: 1-5 (5 = days; 1 = minutes)
- **WSJF priority**: COD ÷ JS (higher = do sooner)

**Main table sort**: by B-ID (insertion-event order, NOT strictly monotonic-by-number). During the Round 3 D72 deep validation cycle, B-numbers were added in batches and inserted at the anchor of the prior closing batch, causing B53-B73 to appear before B38-B50 in the table. The ID itself remains strictly monotonic (every B-number unique and assigned in order of issuance); the table ordering merely reflects when each batch was appended. **B74 tracks the optional re-sort to strict-monotonic-by-B-ID view** (low priority — not load-bearing for any downstream reader). For WSJF-priority view, see "High / Medium / Lower priority" sections below.

**WSJF policy**: update WSJF when scope or context changes; never re-sort silently — log the change.

**Status-render convention** (clarified 2026-05-11 per Pattern F unscoped audit finding):

When a B-item entry shows a leading status badge (e.g., `(🟡 Open)`) AND an inline `**CLOSED YYYY-MM-DD**` annotation in the same row, **the inline annotation is the authoritative status**. The leading badge is sometimes left stale during round close-out when the producer adds the closure annotation inline rather than flipping the badge. This is a known render-discipline gap tracked as B144 (candidate Pitfall #9 sub-class 9.j — B-item status-render discipline) for Round 7 close-out cleanup. Until then, readers should consult inline annotations + the "Closed in Round N" sections (e.g., L242+ for Round 6 closures) as the canonical view, not the leading badge alone.

## How items get added

Two paths:

1. **Validation log → Backlog**: when a `_validation_log.md` entry has 🟡 follow-ups, those items get B-numbers and land here.
2. **Phase deliverable → Backlog**: open Phase deliverables (especially Phase 0's 0.X items that are still ⬜) become B-numbered when they're worked on.

Process:
