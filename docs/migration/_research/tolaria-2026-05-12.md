## Summary

`refactoringhq/tolaria` is a cross-platform (macOS / Windows / Linux) desktop application for managing large markdown-based knowledge vaults, built on Tauri v2 (Rust) + React 19 + TypeScript. Its core proposition is "files-first, git-native": every vault is a folder of plain `.md` files version-controlled by git, with no proprietary format and no cloud dependency. The application is explicitly AI-first — it exposes 14 MCP tool calls to spawned CLI agents (Claude Code, Codex, OpenCode, Gemini, Pi) and supports direct API providers in chat mode — but it is a single-agent-per-session tool, not a multi-agent orchestration framework. It has 9.3k–10.4k stars, 13 contributors, AGPL-3.0 license, and was releasing daily builds as recently as 2026-05-12. The creator uses it to manage a 10,000+ note personal vault and built the entire 100K-line codebase using AI assistance. Confidence in this characterization is high — primary source is the GitHub repo itself plus the creator's launch article.

## Findings

### Finding 1: What Tolaria is — a markdown knowledge-base desktop app, not an orchestration framework

- Source: [1], [2], [4]
- Summary: Tolaria is a knowledge management desktop application. It stores notes as plain `.md` files with YAML frontmatter in a git-backed folder called a "vault." The architecture is three-layer: filesystem (source of truth) → git-based cache (`~/.laputa/cache/`) → React in-memory state. The app never owns the data — it only reads and writes files.
- Relevance to our project: Our project already has a markdown-heavy, git-native document discipline across `docs/migration/` — the same pattern Tolaria is built around. Tolaria is an application FOR managing that kind of structure, not a framework that replaces it.
- Confidence: High

### Finding 2: AI integration footprint — MCP server + single CLI agent per session

- Source: [4], [3]
- Summary: Tolaria exposes a built-in MCP server offering 14 vault-scoped tools: `open_note`, `read_note`, `create_note`, `search_notes`, `edit_note_frontmatter`, `delete_note`, `link_notes`, etc. CLI agents (Claude Code, Codex, OpenCode, Pi, Gemini) are spawned as subprocesses with vault-scoped access. Permission modes (Safe = read-only / Power User = write + shell) gate what agents can do. The AI panel supports one agent session at a time — this is explicitly single-agent, not multi-agent.
- Relevance to our project: Our project uses Claude Code (`.claude/agents/`, `.claude/skills/`) with a multi-agent Pattern E / Pattern F discipline. Tolaria's MCP integration would expose our `docs/migration/` vault to a Claude Code subprocess — functionally similar to how our agents currently read files via the `Read` tool. Tolaria does not add multi-agent orchestration, validation gating, D-number decision tracking, or the 5-gate CHECKS_AND_BALANCES discipline.
- Confidence: High

### Finding 3: No validation workflow, audit trail, or cross-reference enforcement

- Source: [4]
- Summary: Tolaria's architecture documentation explicitly states there are no multi-step validation workflows. Cross-document references use wikilinks (`[[target]]`), which are navigational only — no constraint enforcement, no referential integrity checks, no gate-gating of status flips. Git history is exposed via "Pulse View" but the architecture does not describe an append-only audit log comparable to our `_validation_log.md`.
- Relevance to our project: Our project's most load-bearing disciplines (D55 5-gate validation, D56 mandatory second-pass, Pattern F cascade audit, `_validation_log.md` append-only audit trail) have no analog in Tolaria. Tolaria would not strengthen or replace these disciplines.
- Confidence: High

### Finding 5: Integration footprint — desktop application, not a library or service

- Source: [1], [3]
- Summary: Tolaria is a standalone desktop GUI application installed on a workstation. It is built with Tauri (Rust + WebView) and distributed as a native binary. There is no Python library, no REST API, no CLI-only mode, and no headless mode. Using it requires installing the desktop app and pointing it at a folder of markdown files. The project's AGPL-3.0 license means any derivative work must also be AGPL — relevant if integration involves forking or embedding.
- Relevance to our project: Our pipeline runs on RHEL Linux servers without desktop GUI access. The dev workstation (Windows 11, confirmed from env context) could run the desktop app, but the pipeline planning docs at `docs/migration/` are already perfectly readable by Claude Code agents via direct file reads. Tolaria would add a GUI layer on top of what already works without it.
- Confidence: High

## Recommendation

**Tangential to this project's planning and AI-assisted cascade workflow. No integration action recommended at this time.**

Tolaria (`refactoringhq/tolaria`) is a well-executed markdown knowledge management desktop app with genuine AI-agent integration (MCP tools, Claude Code support, git-native). However, it does not provide:

- Multi-agent orchestration (Pattern E / Pattern F equivalents)
- 5-gate validation discipline enforcement (D55)
- Append-only audit trails comparable to `_validation_log.md`
- D-number / B-number / R-number tracking
- Cross-reference constraint enforcement (wikilinks are navigational only)
- Any CLI or library API suitable for embedding in a pipeline or RHEL server workflow

Our project's document discipline already IS the kind of markdown-plus-git knowledge base Tolaria is built to manage. Adding Tolaria as a GUI viewer on the engineer's Windows workstation would be a personal productivity tool choice — harmless, but not a planning-discipline or AI-workflow decision. The parent agent should record this as a personal-tooling option rather than an architectural integration.

**If the user's interest is specifically in the MCP vault-tool integration pattern** (14 tools Claude Code can call to read/write notes), that pattern is worth noting for Phase 5+ tooling reviews — not for the current Phase 2 pilot work.

---

## Counter-evidence

The one argument for Tolaria integration: its AGENTS.md / CLAUDE.md convention for giving AI agents "vault-aware" context is structurally similar to our own `CLAUDE.md` + CCL discipline. If the user wants a GUI for browsing `docs/migration/` files with wikilink navigation and AI-agent spawning, Tolaria provides that. No external source argues Tolaria is inappropriate for this use case — the counter-evidence to adopting it is simply that the problem it solves (markdown file navigation) is already solved by the existing Claude Code `Read` tool + the `docs/migration/` document set. Tolaria adds GUI ergonomics, not capabilities.
