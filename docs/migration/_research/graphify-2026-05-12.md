## Summary

`safishamsi/graphify` is an open-source Python AI coding-assistant skill that maps any folder — code, SQL schemas, shell scripts, markdown documentation, PDFs, images, and videos — into a queryable knowledge graph. It reached 47.2k GitHub stars with 5.1k forks as of 2026-05-12, making it a high-traction community tool. The tool runs locally: no cloud dependency, no data leaves the machine. It integrates natively with Claude Code by writing a CLAUDE.md instruction block and installing a PreToolUse hook that fires before every Glob/Grep call, surfacing the knowledge graph to Claude before any file search. Output is three artifacts: an interactive HTML graph, a GRAPH_REPORT.md summary (god nodes, surprising cross-file connections, suggested query questions), and a queryable graph.json exposed as an MCP server. The project is maintained primarily by Safi Shamsi, MIT-licensed, and was releasing multiple versions per day in May 2026, indicating active rapid development. For our project's context, Graphify is **partially relevant**: it can ingest our `docs/migration/` markdown planning corpus and surface cross-document structural connections that our current grep-based approach misses. However, it does not provide validation discipline, audit-trail enforcement, D-number/B-number constraint checking, or any equivalent to our 5-gate CHECKS_AND_BALANCES discipline. Confidence in this characterization is high — primary sources are the GitHub repo, release history, and official documentation.

## Findings

### Finding 1: What Graphify is — a multi-modal knowledge graph builder for AI coding assistants

- Source: [1], [2]
- Summary: Graphify is a Python CLI tool that runs locally and processes a project folder in three passes. Pass 1: deterministic AST analysis via Tree-sitter (29 programming languages) extracts function calls, imports, docstrings, and call graphs. Pass 2: local transcription of multimedia via faster-whisper. Pass 3: parallel Claude subagents extract semantic relationships from markdown, PDFs, and images — outputting nodes and edges per batch. NetworkX assembles the full graph; Leiden clustering identifies communities. The result is a `graph.json` (queryable), `graph.html` (interactive browser), and `GRAPH_REPORT.md` (human-readable highlights).
- Relevance to our project: Our `docs/migration/` folder is 100% markdown — no code. Graphify's Pass 3 is the relevant path: Claude subagents read markdown files in batches and extract semantic relationship edges. This could surface cross-document linkages (e.g., which decisions a given edge case mitigation cites, which runbooks reference a given risk) as graph traversal instead of grep. The GRAPH_REPORT.md output is analogous in concept to our HANDOFF.md summary but generated automatically from content rather than maintained by hand.
- Confidence: High

### Finding 2: Claude Code integration — PreToolUse hook + CLAUDE.md injection

- Source: [1], [2], [8]
- Summary: Running `graphify install` writes two artifacts into the project. (a) A section appended to `CLAUDE.md` instructing Claude to read `graphify-out/GRAPH_REPORT.md` before answering architecture questions. (b) A PreToolUse hook entry in `.claude/settings.json` that fires before every `Glob` and `Grep` call — surfacing the knowledge graph hint to Claude before raw file search. The hook message tells Claude: "Knowledge graph exists. Read GRAPH_REPORT.md for god nodes and community structure before searching raw files."
- Relevance to our project: Our project already maintains a CCL (Canonical Context Load) discipline that mandates reading NORTH_STAR.md + HANDOFF.md + CURRENT_STATE.md + CHECKS_AND_BALANCES.md before any other tool call. The Graphify PreToolUse hook would fire at the same layer — before Glob/Grep calls. Conflict risk: if Graphify's hook modifies `.claude/settings.json` and our existing `permissions.deny` rules are in that same file, the hook installation needs to be verified for additive-only behavior before use. This is a caution, not a blocker — it is additive.
- Confidence: High

### Finding 3: MCP server and graph query interface

- Source: [2], [3]
- Summary: Graphify exposes `graph.json` as an MCP server via `python -m graphify.serve graphify-out/graph.json`. Four MCP tool calls are available: `query_graph`, `get_node`, `get_neighbors`, and `shortest_path`. Terminal graph commands include `/graphify query`, `/graphify path`, and `/graphify explain`. The `shortest_path` tool is particularly relevant: it answers "what is the connection between node A and node B?" — analogous to "what decisions does edge case P0-1 trace back to?"
- Relevance to our project: Our current cross-document reference tracing is manual (grep for D-numbers, B-numbers, RB-numbers in decision docs, edge case registers, runbooks). A graph with `shortest_path` queries could surface traceability paths automatically — e.g., "how does R-number R06 connect to the CDC engine module?" This aligns with the Traceability pillar of NORTH_STAR. However, Graphify's semantic edges are LLM-inferred with confidence tags (EXTRACTED / INFERRED / AMBIGUOUS) — they are not guaranteed to be correct. The AMBIGUOUS category in particular requires human verification before treating graph edges as authoritative cross-references.
- Confidence: High (on capability); Medium (on accuracy of inferred edges for our planning docs)

### Finding 4: Documentation-specific behavior — markdown as first-class input

- Source: [3], [2]
- Summary: Markdown files are processed in Pass 3 alongside PDFs and images. Claude subagents read batches and output structured JSON: nodes with `file_type: document`, and edges with `relation` (e.g., `semantically_similar_to`) and confidence scoring. Graphify also exports a `--wiki` markdown wiki and can create an Obsidian vault from the graph. Cross-document references within wikilinks (`[[target]]`) are navigational only — no referential integrity enforcement. Graphify does not detect our specific cross-reference format (D-numbers, B-numbers, RB-N notation), though LLM extraction in Pass 3 may surface them as semantic edges if the prompts encounter them.
- Relevance to our project: Our `docs/migration/` uses a dense cross-reference system (D-numbers, B-numbers, R-numbers, RB-N, SP-N, Pitfall-N, Pattern A-F) that is structurally richer than standard wikilinks. Graphify's LLM extraction would see these as text patterns and may or may not build correct graph edges from them. There is no documented support for custom cross-reference formats. This is the primary gap: Graphify generates semantic similarity edges, not schema-validated constraint edges. Our `_validation_log.md` append-only discipline and the 5-gate CHECKS_AND_BALANCES have no analog in Graphify's output.
- Confidence: High

### Finding 6: Integration footprint — lightweight, local, no infra required

- Source: [1], [2]
- Summary: Graphify is a Python CLI tool installed via pip/uv. It runs entirely locally — no cloud API calls for graph construction (LLM calls use the local coding assistant's API, which Claude Code is already authenticated to). No daemon, no persistent service, no database server. The MCP server mode (`python -m graphify.serve`) is optional and runs in-process. Git hook integration (`graphify hook install`) is optional. The `graphify-out/` folder is the only output artifact.
- Relevance to our project: The RHEL Linux production server has no desktop GUI and Claude Code is not installed there (per D103). Graphify would run on the Windows dev workstation (or dev RHEL workstation) against the `docs/migration/` folder. There is no conflict with the D103 security model — Graphify does not access credentials or credential paths. The `graphify install` step modifies `CLAUDE.md` and `.claude/settings.json`, both of which are inside the `/debi` working-directory boundary.

## Recommendation

**Partially relevant — credible tool for navigation-layer augmentation, but not a planning-discipline replacement. Pilot warranted on a low-stakes basis; no architectural decision needed yet.**

`safishamsi/graphify` is a genuine, high-traction tool (47.2k stars, MIT, local, Claude Code native) that solves a real navigation problem: finding structural connections across a large multi-file corpus without brute-force grep. For our `docs/migration/` planning corpus (~30+ markdown files, dense D/B/R/RB cross-references), Graphify could produce a graph that surfaces "which documents reference Decision D55" or "what is the shortest conceptual path between NORTH_STAR.md and edge case P0-1?" faster than manual search.

However, Graphify does NOT provide:

- Validation discipline enforcement (no 5-gate CHECKS_AND_BALANCES equivalent)
- Append-only audit trail (no `_validation_log.md` equivalent)
- Constraint-validated cross-references (graph edges are LLM-inferred with EXTRACTED / INFERRED / AMBIGUOUS confidence, not schema-enforced)
- D-number / B-number format awareness (standard semantic extraction only)
- Multi-agent orchestration (Pattern E / Pattern F equivalents)
- Status tracking (no 🟡/🟢/🔴 lifecycle management)

The appropriate framing is: Graphify is a navigation-layer tool, not a planning-discipline tool. It would augment how Claude Code discovers cross-document connections (via the PreToolUse hook + GRAPH_REPORT.md), not replace or strengthen the validation discipline that governs status flips and artifact correctness.

**Specific caution**: the PreToolUse hook writes to `.claude/settings.json`. Before installing, verify it is additive-only and does not overwrite existing `permissions.deny` entries (per D103 security model). The hook installs a "before Glob/Grep" trigger — our CCL discipline already mandates a specific read order before any tool call. The two approaches are compatible but the interaction should be tested on a dev workstation first, not production docs.

**If the user wants to pilot it**: run `graphify` against `docs/migration/` on the Windows dev workstation, review `GRAPH_REPORT.md` for surfaced connections, and compare against what grep already finds. One afternoon of pilot work would confirm whether the LLM-extracted edges for our custom cross-reference format are accurate enough to be useful.

## Counter-evidence

The one argument against using Graphify: the LLM-extracted semantic edges for planning docs are confidence-tagged INFERRED or AMBIGUOUS for relationships Graphify cannot deterministically extract. Our project's cross-references use a custom format (D-numbers, B-numbers, RB-N) that Tree-sitter's AST analysis (Pass 1, for code) does not understand, meaning all `docs/migration/` graph edges rely on Pass 3 Claude subagent extraction — a probabilistic step. If the extracted graph has incorrect edges (e.g., asserting that NORTH_STAR.md "references" BACKLOG.md in a way that is semantically wrong), Claude Code following the PreToolUse hint could navigate to the wrong document. Our current grep-based approach is 100% accurate for D-number cross-references (they are literal strings). The counter-argument to adopting Graphify is: our existing literal grep already handles structured cross-references correctly; Graphify adds semantic discovery at the cost of introducing LLM inference error into navigation. For planning docs with a dense structured cross-reference schema, the literal approach may be more reliable than graph traversal.

No external source explicitly argues Graphify is inappropriate for documentation-only repos — the counter-evidence is structural (inference error risk) rather than documented.
