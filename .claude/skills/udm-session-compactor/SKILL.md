# UDM Session Compactor

Manual-trigger snapshot authoring skill that captures session state at higher fidelity than `SESSION_RESUME.md` — preserves architectural decisions, rejected alternatives, cross-cohort patterns, and why-not justifications so a new-context-window agent (post-compaction OR post-session-pause) can resume with deeper insight than the canonical 5-tracker minimum (BACKLOG + CURRENT_STATE + HANDOFF + CODE_BUILD_STATUS + _validation_log).

## Why this skill exists (the gap it closes)

**Empirical evidence base (1-event 2026-05-18)**: this session underwent context-window compaction during the arc. The pre-compaction summary (Claude Code system-generated) captured WHAT happened but lost WHY — architectural decisions, rejected alternatives, cross-cohort patterns, and the cumulative reasoning behind 99 NEW B-Ns / 23 B-N closures. `SESSION_RESUME.md` proved insufficient as a recovery anchor:

1. **Stale-arithmetic class** — gap-check reviewer `ab45539c33d1cebd1` G2 finding 2026-05-18: SESSION_RESUME.md L10-L15 cited `98 NEW B-Ns / 17 CLOSED / 2798 pass / latest commit abb7596` for 2-3 cohorts AFTER actual state advanced to `99 / 23 / 2817 / c781c9b`. Manual refresh missed multiple session-events.
2. **Insight-loss class** — by design, SESSION_RESUME.md is a brief pointer (last-commit + cumulative counts + arc summary). It captures the WHAT but does NOT capture the WHY — e.g., "why did we reject Option B JSON-sidecar baseline for B-486 in favor of Option A env-configurable threshold?" or "what's the architectural insight behind the 4-layer false-positive prevention architecture?" Those rationales live ONLY in the conversation transcript, which is exactly what compaction sheds.
3. **Cross-compaction-event class** — post-compaction agent has access to (a) compaction summary auto-generated by Claude Code + (b) on-disk artifacts. The compaction summary describes mechanical action sequences; it does NOT describe judgment calls. Pre-compaction snapshot fills this gap.

**Failure-mode classes the snapshot mitigates**:

| Class | Pre-snapshot state | Post-snapshot state |
|---|---|---|
| Why-not loss | "We decided X" without justification | "We decided X over Y because Z" |
| Cross-cohort pattern erosion | Patterns scattered across 5-7 commits | Pattern explicitly named + commit-anchored |
| Architectural-seed forgetting | Future-state architectural concerns marked as deferred B-N but rationale lost | Rationale preserved alongside B-N pointer |
| Convergence-discipline drift | "Why did we converge on this skill design?" | Convergence path explicit |
| Reviewer-finding context | Reviewer agent IDs cited but methodology / verdict-detail lost | Verdict + methodology + delta-from-prior-reviewer captured |

**Distinction from existing layers**:

| Layer | Scope | Cadence | Output |
|---|---|---|---|
| `udm-progress-logger` (hard rule 9) | Per-completion canonical-tracker discipline | After each substantive work | BACKLOG + CURRENT_STATE + HANDOFF + _validation_log row updates |
| `SESSION_RESUME.md` | Session-pause pointer | At session pause (manual refresh) | Single-file last-commit + counts + runway |
| **`udm-session-compactor` (this skill)** | **Deep insights + WHY behind decisions across session arc** | **Manual-trigger OR pre-compaction OR pre-SESSION_RESUME write** | **`_session_snapshots/<date>-<commit>.md` 5-section artifact** |
| `udm-cohort-review` (hard rule 11 extension) | Cross-cohort drift detection across 2-N commits | User-invokable / pre-SESSION_RESUME-write | Verdict report |
| `udm-round-closeout` (D60) | Round-aggregate cascade | Round close-out only | 5-tracker aggregate updates |

- "compress the session" / "compress this session"
- "snapshot the session" / "take a session snapshot"
- "session compactor" / "run session compactor"
- "preserve deeper insights" / "preserve context before compaction"
- "save session state" / "save the session"

**Auto-invocable triggers** (parent agent SHOULD invoke without user prompt):

- **Before SESSION_RESUME.md write** at session pause/close (snapshot AUGMENTS resume doc with deeper insights; the resume doc remains the pointer; the snapshot is the substrate)
- **Before any anticipated context-compaction event** when agent detects (a) ≥10 substantive commits in session arc OR (b) ≥3 new skills/agents authored OR (c) ≥1 architectural-layer addition (e.g., new false-positive prevention layer; new check abstraction; new discipline hard rule)

**NOT auto-invocable triggers**:

- Per-commit auto-trigger — duplicates `udm-progress-logger`
- Round-close-out trigger — duplicates `udm-round-closeout`
- Trivial-cohort sessions (e.g., 1-2 cosmetic commits; no architectural work) — overhead exceeds value

## Anti-triggers (do NOT invoke even if a trigger-like phrase appears)

- User asking a meta-question about the skill ("what does session-compactor do?") — explain don't invoke
- Session is fresh / no architectural work has accumulated — defer
- User asking about token tracking / context-window mechanics — that's claude-code-guide research scope (Phase 2 deferred)
- User invoking on per-commit cadence — that's `udm-progress-logger` scope
- User invoking at round close-out — that's `udm-round-closeout` scope
- Compaction has ALREADY occurred and current turn is post-compaction — snapshot now would capture post-compaction state (lossy); instead read the compaction summary + existing trackers

## The 5-section snapshot procedure (canonical)

For each invocation: parent agent identifies the session arc scope (last N commits OR commits since last `_session_snapshots/` artifact) + authors the snapshot directly (no sub-agent spawn — this is a producer-side discipline; verification happens at gap-check separately).

The snapshot file lives at `docs/migration/_session_snapshots/<YYYY-MM-DD>-<commit-hash-prefix-7>.md` with this canonical structure:

### §1 — Active work context

- Current branch name + base branch + commits ahead
- Current B-Ns in flight (open, unclosed) with WSJF + closure target
- Current cohort scope (what theme is the session arc executing? e.g., "Phase 1 commit-msg ABC abstraction" or "False-positive prevention architecture")
- Pipeline-lead direction at session start (verbatim quote when available)
- Last-known stable state (e.g., last-passed pytest count + commit hash)

### §2 — Completed deliverables

- B-Ns closed in session arc (full enumeration; mechanism per closure; closure mechanism — e.g., "single-commit substrate-edit at <path>" or "absorbed via <other-B-N>")
- Commits in session arc with hashes + 1-line summaries
- Test count delta (baseline → current)
- New public surface added (functions / classes / constants / EventTypes — full enumeration per Step 10 / Pitfall #9.n)
- New skills / agents / canonical trackers authored
- Convention-registrations applied (CLAUDE.md Structure rows + GLOSSARY entries + CLI_* family registry updates)

### §3 — Open runway

- Deferred B-Ns with explicit defer-trigger criteria (e.g., "defer until empirical 2nd-event")
- LOW WSJF items not yet executed (enumerated to avoid forgetting)
- Pipeline-lead-blocked queues (Phase 2-4 UDM Skills Audit cluster B-416-B-447; etc.)
- POLISH_QUEUE P-N items still open
- Known-but-not-tracked patterns (with reasoning for not opening B-N yet)

### §4 — Deeper insights (THE differentiator vs SESSION_RESUME.md)

This is the section that distinguishes the snapshot from the lightweight resume pointer. Capture:

- **Architectural decisions made**: each architectural call with explicit (Decision / Alternatives considered / Why-not for each alternative / What would trigger reversal)
- **Cross-cohort patterns surfaced**: patterns visible only across multiple commits (e.g., "4-layer false-positive prevention architecture emerged across B-488 + B-489 + B-490 + B-481 cohort because <reasoning>")
- **Convergence-discipline events**: where the session converged on a discipline rule via multiple iterations (e.g., "B-459 ABC abstraction landed only after B-470/471/472 polish iterations because <reasoning>")
- **Rejected alternatives with rationale**: e.g., "B-486 Option B JSON-sidecar deferred because <reasoning>; revisit if Option A proves insufficient at <empirical trigger>"
- **Reviewer-finding context**: independent reviewer agent IDs + verdicts + key findings + what changed because of each finding
- **Empirical-evidence accumulations**: pattern-anchor commits + event count + threshold for formalization (e.g., "Pitfall #9.h is 1-event class; awaiting 2nd-event before HANDOFF §8 formalization")
- **Meta-discipline observations**: e.g., "single-commit reviewers cannot see cross-cohort interactions by construction → motivated B-483 udm-cohort-review skill"

### §5 — Pointer-back cross-refs

- Canonical trackers that should be read by next-window agent (with current line-anchors for fast navigation)
- Recent commit hashes for `git show` reference
- Skill files relevant to session arc
- Open `_research/` documents relevant to active work
- BACKLOG.md sections to scan
- _false_positive_log.md if false-positive prevention work was substantive

## Content durability — regenerable vs irreplaceable (trim-policy taxonomy)

Per research finding 2026-05-18 (Contextual Memory Virtualisation paper, Imperial College London; `_research/llm-handoffs-traceability-hallucination-2026-05-18.md` Finding 1.1): naive compaction destroys ~98% of nuanced session reasoning while preserving only surface summary. The CMV alternative achieves 12-86% reduction (median 20%) by preserving every irreplaceable element verbatim and discarding only regenerable raw output. This taxonomy operationalizes that distinction for snapshot authoring.

### Regenerable content (MAY be omitted or compressed in snapshots)

Content that can be reconstructed mechanically from primary docs / git / file system. Authoring agents SHOULD omit verbatim and reference by pointer:

- Raw `grep` / `Glob` / file-listing outputs (re-run anytime via Bash tool)
- `git log --oneline` / `git diff --stat` outputs (re-derivable from git)
- Verbatim test stdout (re-derivable via `pytest`)
- Raw file content quotations (cite path + line number; re-readable via Read tool)
- Tool-call success confirmations / boilerplate progress updates
- Verbatim BACKLOG.md entries (cite B-N + line number; re-readable)
- Tracker-update mechanics ("propagated 99→100 NEW B-Ns" — count is in tracker; re-readable)

### Irreplaceable content (MUST be preserved verbatim or in structured form)

Content that exists ONLY in the conversation transcript and is lost on compaction. Authoring agents MUST preserve verbatim:

- **Architectural decisions + rejected alternatives + why-not justifications** — these are NOT in commit messages or D-N bodies; they exist only in conversation
- **Reviewer rationale text** (verbatim quotation of reviewer agent verdicts + key findings + delta-from-prior-reviewer) — agent IDs are pointers; the reasoning behind verdicts is irreplaceable
- **Empirical-evidence accumulations** (event count + commit anchors + threshold-status for formalization) — the meta-pattern aggregation is irreplaceable; individual commits are recoverable
- **Pipeline-lead direction at session start / pivot points** (verbatim user-direction quote) — context for WHY work happened
- **Cross-cohort pattern observations** — patterns visible only across multiple commits; would require re-deriving from full session transcript
- **Convergence-discipline events** — multi-iteration discovery of correct discipline rule; rationale is irreplaceable
- **Defer-trigger criteria for B-Ns** — the "when to come back" condition that determines whether a deferred B-N stays deferred
- **Meta-discipline observations** — emergent patterns about how the project's discipline stack works

### Trim-policy enforcement (producer-side check)

When authoring §1-§5 of a snapshot:
- For each bullet/paragraph: ask "can this be re-derived from primary docs + git + file system?" If YES → consider omitting + replacing with pointer. If NO → preserve verbatim.
- Bias toward preservation when in doubt — under-trimming is recoverable (re-trim at next snapshot); over-trimming is lossy (irreplaceable content is gone).
- §4 Deeper insights is structurally biased toward irreplaceable content; aggressive preservation expected.
- §1 Active work + §2 Completed deliverables + §3 Open runway + §5 Pointer-back cross-refs are structurally mixed; apply per-bullet judgment.

### Empirical anchor for trim-policy

1-event 2026-05-18: this session's compaction caused 2-3 cohort lag in SESSION_RESUME.md L10-L15 cumulative-count drift (surfaced by gap-check reviewer `ab45539c33d1cebd1` G2 finding 2026-05-18). The information lost was IRREPLACEABLE (WHY the count was at a specific state; what reviewer findings drove the increments). The CMV paper confirms this is the systematic failure mode of naive compaction. The trim-policy formalized here closes the failure-mode class at producer-time before snapshots are authored.

## Output contract

The snapshot file at `docs/migration/_session_snapshots/<YYYY-MM-DD>-<commit-hash-prefix-7>.md` MUST contain:

- Top-level YAML frontmatter `--- snapshot_date / commit_hash / session_arc_scope ---` (3 fields)
- All 5 canonical section headers (`## §1 ` through `## §5 `)
- Minimum 1 entry per section (else the snapshot is premature)
- File size ≥ 2KB (per B-558 Phase 2.1 Component B 2026-05-19 — `.claude/hooks/session-compactor-warning.py::_is_structurally_valid_snapshot()` mechanically enforces this floor; stub-class snapshots do NOT suppress the auto-trigger warning)
- Footer with pointer to next-snapshot trigger criteria (when to refresh)
- **§6 verification footer** (per B-558 Phase 2.1 Component B closure 2026-05-19 post-authoring verification mandate): single-line citation of `udm-gap-check` invocation + reviewer agent-ID + verdict. Format: `**Verification**: udm-gap-check reviewer <agent-id> verdict ✅ CLEAN / 🟡 IN-FLIGHT / 🔴 BLOCK on YYYY-MM-DD`. Absence of §6 indicates snapshot was authored WITHOUT post-authoring verification — discipline-debt class.

After authoring, parent agent:

1. Writes a `_validation_log.md` event row referencing the snapshot file
2. Adds a row to BACKLOG.md (or POLISH_QUEUE.md if cosmetic) IF the snapshot surfaced new tracking-worthy items not already tracked
3. **Refreshes `SESSION_RESUME/active/<chat-name>.md` state pointer** for THIS chat per B-562 Component B multi-chat coordination cohort (closed 2026-05-19). Update sections: state-as-of-session-end (branch / latest commit / push status / pytest baseline / cumulative session delta) + this-session commit chain + NEXT SESSION RESUME PROCEDURE + Open runway (priority-ordered). Do NOT touch other chats' active files. See `SESSION_RESUME/README.md` for chat-naming convention + lifecycle (active → _archive on clean session end). If `SESSION_RESUME/active/<chat-name>.md` does not yet exist for THIS chat: author it now (with chat-scope description + initial state); see `SESSION_RESUME/active/meta-discipline.md` as the canonical template
4. **Invokes `udm-gap-check` (per B-558 Phase 2.1 Component B closure 2026-05-19 post-authoring verification mandate)** — spawns independent reviewer agent (per D55+D56 producer ≠ reviewer) to audit snapshot completeness against the 5-section contract. 6-category audit adapted for snapshot scope: G1 N/A (no leading-badge in snapshots) / G2 claims-vs-actual (verify cited commit hashes + pytest counts + B-N closure counts match actual repo state) / G3 cross-refs resolve (all cited paths exist) / G4 discipline applied (§5 cross-refs comprehensive) / G5 N/A (Convention-registration is producer-time Step 10, not snapshot-time) / G6 surfaced patterns (any new B-N candidates flagged in §4 deeper insights). Append §6 verification footer with reviewer agent-ID + verdict immediately after gap-check completes. 🔴 BLOCK verdict → fix-inline + re-spawn reviewer; 🟡 IN-FLIGHT → fix-inline OR escalate per D56 second-pass; ✅ CLEAN → proceed to Step 5.
5. Reports to user: snapshot path + 1-line summary of what's preserved beyond what `SESSION_RESUME/active/<chat-name>.md` captures + gap-check verdict

## Edge cases

- **Empty session arc** (no commits since last snapshot): emit ✅ N/A; do not author empty snapshot
- **Compaction already occurred**: do NOT retroactively author a snapshot — the substrate to compress is lost; instead update SESSION_RESUME.md + author from current trackers
- **Pre-snapshot reviewer findings unresolved**: snapshot can be authored but MUST note open findings in §4; resolve at next cohort
- **Snapshot file path collision** (multiple snapshots same day): append `-2` / `-3` suffix; multiple per-day snapshots are valid for very long sessions
- **Tracker file unavailable** (e.g., BACKLOG.md mid-edit): defer snapshot authoring until trackers are stable
- **Branch already merged to main**: snapshot remains useful for archaeology; author on the branch-as-of-merge

## Do NOT include in snapshots (CCPA/PII compliance per B-559 closure 2026-05-19)

Snapshots are committed to the git repository + may be pushed to GitHub + may be cached by reviewers + downstream tooling. Treat them as **publicly-visible audit artifacts**. Per D102 + D103 + R36 (Phase A plaintext-PII compensating controls):

### NEVER include in snapshots

- **PII values from production tables** (SSN / email / phone / DOB / member-id / account-numbers / credit-card numbers / driver-license / passport / etc.). Even single-row examples are forbidden.
- **Plaintext credentials**: API keys / OAuth tokens / database passwords / GPG private keys / RSA private keys / TPM2-sealed material / Snowflake RSA keys / Anthropic API keys / GitHub PATs / etc.
- **Connection strings with embedded passwords** (e.g., `postgresql://user:password@host/db`). Always redact to `postgresql://user:****@host/db` or omit the URL entirely.
- **Extraction-time row counts AT SUB-DAY granularity** that could de-anonymize a small cohort (e.g., "extracted 7 ACCT rows from 2026-05-15 14:30:00" — if cohort is <100, treat as PII-adjacent).
- **Customer / supplier / merchant names** from production data, even as illustrative examples.
- **GPS coordinates / IP addresses / device IDs** linkable to individuals.

### Safe to include

- B-N / D-N / R-N / RB-N / SP-N references (canonical project codes)
- Commit hashes / branch names / file paths (architectural context)
- Reviewer agent IDs (synthetic identifiers; not PII)
- Test count deltas / Tier 0 PASS counts (aggregate numerics)
- Decision rationale / architectural alternatives / rejected approaches (the §4 deeper-insights substance)
- Pipeline-lead direction quoted verbatim WHEN it does not contain PII

### Operator workflow when in doubt

1. **Pause** before authoring the snapshot
2. **Grep** the proposed content for: `\b\d{3}-\d{2}-\d{4}\b` (SSN-shaped) / `[\w.+-]+@[\w-]+\.[\w.-]+` (email-shaped) / `\b\d{13,19}\b` (credit-card-shaped) / `-----BEGIN (RSA |OPENSSH |EC |PGP )?PRIVATE KEY-----` (key headers) / `[A-Za-z0-9_]{32,}` (high-entropy tokens)
3. **Redact** any matches OR omit the relevant content
4. If unsure, **prefer omission** — the value of session-state preservation does not exceed compliance risk

Mechanical layer: Tier 0 test `tests/tier0/test_session_snapshot_pii_scrub.py` (B-559 closure 2026-05-19) greps all `docs/migration/_session_snapshots/*.md` files for sensitive patterns at every test-suite run; flags any matches as test failures. This is a defense-in-depth layer; the primary discipline is operator-side at authoring time.

## Auto-trigger awareness (Phase 2 — landed 2026-05-19 per B-494 closure)

When the parent agent sees a stderr warning of the form:

```
⚠️  SESSION COMPACTION APPROACHING (per B-494 udm-session-compactor Phase 2 auto-trigger)
Transcript size: <X> MB (~<Y> estimated tokens; <Z>% of 1M Opus 4.7 context)
Threshold crossed: <pct>%

ACTION REQUIRED: Invoke `udm-session-compactor` skill IMMEDIATELY to author session snapshot...
```

it is emitted by the **PostToolUse hook** at `.claude/hooks/session-compactor-warning.py` (per B-494 closure). The hook fires after every tool call, measures the session transcript JSONL byte size (`~/.claude/projects/<slug>/<session-uuid>.jsonl`), estimates token usage via the 5 bytes/token heuristic (per claude-code-guide research 2026-05-18 Path E hybrid), and compares against the configured threshold (default 70%; override via `UDM_COMPACTOR_THRESHOLD_PCT` env var).

### Required agent response

Per the auto-trigger contract:

1. **Acknowledge the warning** — do not dismiss it; treat as high-priority signal
2. **Pause current work at next safe boundary** — finish the current tool call cycle / commit if mid-cohort
3. **Invoke this skill (`udm-session-compactor`)** to author a snapshot per the 5-section procedure at `docs/migration/_session_snapshots/<YYYY-MM-DD>-<commit-hash-prefix-7>.md`
4. **Resume work after snapshot lands** — the warning is suppressed for the rest of the session once a snapshot file is detected in the directory

### Suppression discipline

The hook writes a suppression marker at `.claude/_session_metrics/<session-id>.compactor-warned` once the warning fires. Subsequent hook invocations in the same session skip warning emission until:
- A snapshot file appears in `docs/migration/_session_snapshots/` modified after session start, AND
- The suppression marker is cleared (which happens at next session start)

This prevents warning-spam on every tool call after threshold-crossing.

### Telemetry

The hook also writes a per-tool-call telemetry row to `.claude/_session_metrics/<session-id>.jsonl` with fields `{ts, tool_name, estimated_tokens, transcript_bytes, threshold_pct, warning_fired}`. This enables retrospective drift analysis if the 5 bytes/token heuristic empirically diverges from ground-truth Token Counting API measurements (deferred to Phase 2.1 per B-494).

### Architectural rationale

Per claude-code-guide research 2026-05-18 (`a4c2c9ae8ebd639e1`): Claude Code hooks **cannot directly invoke Claude** to trigger skill invocation. The mechanism is indirect:

```
PostToolUse hook fires      → measures transcript size
                            → compares against threshold
                            → emits stderr warning if threshold crossed
Claude reads tool result    → sees stderr warning in next turn
Claude (per this skill body) → proactively invokes udm-session-compactor
                            → snapshot lands
                            → hook detects snapshot in subsequent runs
                            → suppresses further warnings
```

This is the **Path E hybrid checkpoint pattern**. Active feature request `anthropics/claude-code#34340` (Claude Code expose `CLAUDE_CONTEXT_USED` env var to hooks) — when landed, this hook can use direct token measurement instead of JSONL byte-size heuristic.
